## 1. The Application Code

### SpringbootappApplication.java

This file was auto-generated by start.spring.io. It contains the main method
that starts the entire Spring Boot application.

```java
@SpringBootApplication
public class SpringbootappApplication {
    public static void main(String[] args) {
        SpringApplication.run(SpringbootappApplication.class, args);
    }
}
```

`@SpringBootApplication` — a compound annotation that does three things at once:
- Enables auto-configuration — Spring Boot automatically configures beans based
  on what dependencies are on the classpath
- Enables component scanning — Spring scans the package and sub-packages for
  classes annotated with `@RestController`, `@Service`, `@Repository` etc.
- Marks this as a configuration class

`SpringApplication.run()` — starts the embedded Tomcat server, loads the Spring
application context, and starts listening for HTTP requests on port 8080.

### HelloController.java

```java
package com.portfolio.springbootapp;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import java.util.Map;

@RestController
public class HelloController {

    @GetMapping("/")
    public Map<String, String> hello() {
        return Map.of(
            "status", "ok",
            "message", "Hello from Spring Boot on AWS Fargate",
            "service", "springboot-app"
        );
    }

    @GetMapping("/health")
    public Map<String, String> health() {
        return Map.of(
            "status", "healthy",
            "service", "springboot-app"
        );
    }
}
```

`package com.portfolio.springbootapp` — every Java file declares its package.
The package name must match the folder structure. This prevents naming conflicts
across large projects.

`import` statements — bring in classes from other packages. Without these Java
does not know what `@RestController`, `@GetMapping`, or `Map` means.

`@RestController` — tells Spring Boot this class handles HTTP requests and all
return values should be automatically serialized to JSON. In Node.js Express you
explicitly call `res.json({...})`. In Spring Boot `@RestController` handles that
conversion automatically using the Jackson library (included in spring-boot-starter-web).

`@GetMapping("/")` — maps HTTP GET requests to the `/` path to this method.
When someone calls `GET /`, Spring routes it here.

`Map.of(...)` — Java built-in that creates an immutable key-value map in one line.
Spring Boot's Jackson serializer converts this to JSON automatically:
`{"status":"ok","message":"Hello from Spring Boot on AWS Fargate","service":"springboot-app"}`

`@GetMapping("/health")` — the health endpoint. The AWS ALB Target Group health
check hits this URL every 30 seconds. If it returns HTTP 200 the task is healthy.
If it does not respond, the task is deregistered from the load balancer and ECS
starts a replacement. This is why the `/health` endpoint exists in every
production application.

### application.properties

```properties
server.port=8080
management.endpoints.web.exposure.include=health,info
management.endpoint.health.show-details=always
```

`server.port=8080` — Spring Boot defaults to 8080. We set it explicitly so it
is clear and intentional. The Task Definition, Target Group, and Fargate security
group all use port 8080. Everything must match.

`management.endpoints.web.exposure.include=health,info` — Spring Boot Actuator
exposes management endpoints. By default most are hidden for security. This
exposes only health and info at `/actuator/health` and `/actuator/info`.

`management.endpoint.health.show-details=always` — makes the actuator health
endpoint return detailed information about every component's health status
instead of just `{"status":"UP"}`.

## 2. pom.xml — The Maven Project Manifest

`pom.xml` is the Maven equivalent of `package.json` in Node.js. POM stands for
Project Object Model. It defines what the project is, what it depends on, and
how to build it.

```xml
<groupId>com.portfolio</groupId>
<artifactId>springboot-app</artifactId>
<version>0.0.1-SNAPSHOT</version>
```

`groupId` — organisation identifier, like a reverse domain name. `com.portfolio`
for this project. Real companies use `com.amazon`, `com.flipkart` etc.

`artifactId` — the project name. This becomes part of the JAR filename:
`springboot-app-0.0.1-SNAPSHOT.jar`

`version` — `SNAPSHOT` means work in progress. When releasing to production you
change this to `1.0.0`.

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.5.14</version>
</parent>
```

`parent` — our project inherits from Spring Boot's parent POM. This parent
contains pre-configured compatible versions of hundreds of libraries. We never
specify individual dependency versions — the parent manages compatibility. Upgrade
the parent version and all dependencies upgrade in a compatible way automatically.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
```

`spring-boot-starter-web` — a starter that bundles everything needed for a web
application: Spring MVC, embedded Apache Tomcat server, Jackson JSON library,
validation libraries. One dependency, everything included.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

`spring-boot-starter-actuator` — adds management endpoints automatically.
`/actuator/health` is the most important one — used by Kubernetes liveness probes,
AWS ALB health checks, and monitoring tools to verify the app is alive.

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
```

`scope>test` — this dependency is only available during testing, never in the
final JAR. Same concept as `devDependencies` in Node.js. JUnit, Mockito, and
other test libraries are included here. Maven's `--omit=dev` equivalent.

```xml
<plugin>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-maven-plugin</artifactId>
</plugin>
```

`spring-boot-maven-plugin` — makes `mvn spring-boot:run` work locally and creates
an executable fat JAR during `mvn package`. A fat JAR contains your compiled code
AND all dependencies bundled inside — so you can run it with just `java -jar app.jar`
without anything else installed. This is what goes into the Docker container.

**Node.js vs Maven comparison:**

| Node.js | Maven |
|---------|-------|
| `package.json` | `pom.xml` |
| `npm install` | `mvn install` |
| `npm test` | `mvn test` |
| `npm start` | `mvn spring-boot:run` |
| `node_modules/` | `~/.m2/repository/` |
| `dependencies` | `<dependency>` without scope |
| `devDependencies` | `<dependency scope="test">` |
| `npm run build` | `mvn package` |

## 3. The Dockerfile — Two Stage Build

```dockerfile
# Stage 1 — Build
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2 — Run
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Stage 1 — Build

`FROM maven:3.9-eclipse-temurin-17 AS builder` — the build environment. This
image contains Maven 3.9 AND Java JDK 17 — everything needed to compile Java.
`AS builder` names this stage so Stage 2 can reference it. This image is ~500MB.

`COPY pom.xml .` then `COPY src ./src` — copied separately for Docker layer caching.
If you change Java code but not `pom.xml`, Docker reuses the cached `mvn install`
layer and skips re-downloading all dependencies. Maven downloading dependencies
can take 2-3 minutes. This ordering saves that time on every rebuild.

`RUN mvn clean package -DskipTests` — compiles all Java files and creates the fat
JAR in `/app/target/`. `-DskipTests` skips running tests because tests already ran
in the GitHub Actions pipeline before Docker build. Running them again wastes time.

### Stage 2 — Run

`FROM eclipse-temurin:17-jre-alpine` — completely fresh start with a minimal image.
JRE only (not full JDK) — can run compiled Java but cannot compile it. Alpine Linux
keeps it tiny. This image is ~85MB compared to 500MB for the Maven image.

`COPY --from=builder /app/target/*.jar app.jar` — the key line. Copies only the
compiled JAR from Stage 1 into Stage 2. Everything else in Stage 1 — Maven, JDK,
source code, downloaded dependencies — is completely discarded.

`ENTRYPOINT ["java", "-jar", "app.jar"]` — runs the JAR when the container starts.
`ENTRYPOINT` vs `CMD` — ENTRYPOINT always runs and cannot be replaced, CMD can be
overridden. For a Java app ENTRYPOINT is correct — you always want `java -jar` to run.

**Result: 89MB production image instead of 500MB+**

The two-stage build is a standard Docker best practice for Java. In interviews
when asked "how do you keep Java Docker images small" — two-stage build with
JRE-only base image is the answer.

## 4. The GitHub Actions Pipeline

### Job 1 — Build and Test

```yaml
- name: Set up Java 17
  uses: actions/setup-java@v4
  with:
    java-version: '17'
    distribution: 'temurin'
    cache: 'maven'
```

`distribution: 'temurin'` — installs Eclipse Temurin JDK, same distribution used
in the Dockerfile. Always use the same JDK distribution in CI and Docker for
consistency — different distributions can have subtle behavioral differences.

`cache: 'maven'` — saves the Maven local repository (`~/.m2`) between pipeline
runs. Maven downloads dependencies to this cache. Without caching, every pipeline
run downloads everything from scratch — 2-3 minutes. With caching, dependencies
are restored from the previous run — pipeline runs in seconds.

```yaml
- name: Build and test with Maven
  run: mvn clean verify
```

`mvn clean verify` runs these Maven lifecycle phases in sequence:
- `clean` — delete previous build output
- `compile` — compile all Java source files
- `test` — run all JUnit tests
- `package` — create the fat JAR
- `verify` — run integration tests and verify the package

If any test fails Maven exits with a non-zero code, the step fails, and
`build-and-push` job never starts.

### Job 2 — Build and Push to ECR

```yaml
- name: Configure AWS credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
    aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    aws-region: ap-south-1
```

Configures AWS CLI on the GitHub Ubuntu machine using credentials stored as
GitHub Secrets. After this step all AWS CLI commands and AWS SDK calls are
automatically authenticated. The actual key values never appear in logs.

```yaml
- name: Login to Amazon ECR
  id: login-ecr
  uses: aws-actions/amazon-ecr-login@v2
```

Logs Docker into ECR so it can push images. `id: login-ecr` gives this step a
name so the next step can read its outputs.

```yaml
- name: Build and push Docker image
  env:
    ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
    ECR_REPOSITORY: springboot-app
    IMAGE_TAG: ${{ github.sha }}
  run: |
    docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
    docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:latest .
    docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
    docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest
```

`steps.login-ecr.outputs.registry` — reads the ECR registry URL from the login
step output. Automatically resolves to your account's ECR URL like
`554716176680.dkr.ecr.ap-south-1.amazonaws.com`. You never hardcode your account ID.

Two tags pushed:
- `:latest` — always points to the newest image
- `:git-commit-hash` — unique tag per commit for traceability and rollback

## 5. AWS Resources — What We Created and Why

### ECR — Elastic Container Registry

Created via AWS Console → ECR → Create repository → Private → name `springboot-app`.

ECR is Docker Hub but owned by AWS and private by default. The registry URL is
automatically your account's ECR endpoint:
`554716176680.dkr.ecr.ap-south-1.amazonaws.com`

You create repositories inside this registry. Each repository holds images for
one application. The pipeline pushes to `554716176680.dkr.ecr.ap-south-1.amazonaws.com/springboot-app`.

**Image tag mutability** — we left it as Mutable. This means the `:latest` tag
can be overwritten with a new image on every push. Immutable would prevent tag
overwriting — useful when you want to guarantee that a specific tag always points
to the same image and can never be replaced.

### VPC and Networking

Created via VPC Console → Create VPC → **VPC and more** option.

The **VPC and more** option is the modern AWS way — it creates the complete
networking setup in one operation instead of creating each piece separately:
- VPC with CIDR `10.0.0.0/16`
- 2 public subnets (one per availability zone)
- 2 private subnets (one per availability zone)
- Internet Gateway attached to VPC
- Route tables — public subnets route to Internet Gateway, private subnets
  route to NAT Gateway
- NAT Gateway in one public subnet

**Why public and private subnets:**
- ALB lives in public subnets — needs to receive internet traffic directly
- Fargate tasks live in private subnets — no direct internet access, more secure
- NAT Gateway allows private subnet resources to make outbound internet connections
  (to pull images from ECR, send logs to CloudWatch) without being directly reachable
  from the internet

### NAT Gateway and Elastic IP

**NAT Gateway** — created automatically when we selected NAT Gateway option during
VPC creation. Placed in a public subnet. Allows Fargate tasks in private subnets
to reach the internet for outbound connections.

**Elastic IP** — AWS automatically creates and attaches an Elastic IP to the NAT
Gateway. An Elastic IP is a static public IP address that does not change. Regular
EC2 and NAT Gateway IPs change when resources are stopped and started. Elastic IP
stays the same permanently.

**Critical deletion note:** When you delete a NAT Gateway the Elastic IP is NOT
automatically released. It remains allocated to your account and AWS charges for
unattached Elastic IPs (~$3.60/month). You must manually release it after deleting
the NAT Gateway. This is a common mistake that leads to unexpected AWS bills.

**Deletion order:** Delete NAT Gateway first → wait for Deleted status → then
Release Elastic IP. If you try to release the IP while the NAT Gateway still exists,
AWS will not allow it.

### Security Groups

**ALB Security Group (`springboot-alb-sg`):**
- Inbound: port 80 from `0.0.0.0/0` (entire internet)
- Outbound: all traffic allowed

**Fargate Security Group (`springboot-fargate-sg`):**
- Inbound: port 8080 from `springboot-alb-sg` only — not from the internet
- Outbound: all traffic allowed

The Fargate security group uses the ALB security group as the source instead of
an IP range. This means only traffic coming from the ALB can reach the Fargate
containers. If someone tries to hit the Fargate task directly with its private IP,
the security group blocks it. This is called **security group chaining** and is
the standard production pattern.

### Application Load Balancer

Created via EC2 Console → Load Balancers → Create → Application Load Balancer.

**Scheme: Internet-facing** — ALB gets a public DNS name accessible from the internet.
The alternative is Internal — only reachable from within the VPC, used for
microservice-to-microservice communication.

**Availability Zones** — we selected both AZs and their public subnets. ALB
distributes traffic across both AZs automatically. If one AZ has issues, ALB
routes all traffic to the other AZ.

**DNS name** — AWS generates a DNS name like
`springboot-alb-50032053.ap-south-1.elb.amazonaws.com`. This is your app's
public URL. In production you create a Route 53 alias record pointing your
custom domain to this ALB DNS name.

### Target Group

Created during ALB creation — **target type must be IP addresses**, not instances.
Fargate tasks do not have EC2 instance IDs. They have IP addresses assigned from
the VPC subnet. Using Instance type would mean ECS cannot register Fargate tasks.

**Health check path: `/health`** — ALB pings this endpoint every 30 seconds.
If a task fails 3 consecutive health checks it is deregistered and ECS starts
a replacement task.

### ECS Cluster

Created via ECS Console → Clusters → Create cluster.

The first time we tried creating the cluster it failed with:
`Unable to assume the service linked role`

This happened because ECS needs a service-linked IAM role called
`AWSServiceRoleForECS` that gets created automatically the first time ECS is
used in an account. Since this AWS account had a previous failed Terraform
experiment that left a broken CloudFormation stack, the role creation was
stuck.

**Fix:** We ran `aws iam create-service-linked-role --aws-service-name ecs.amazonaws.com`
manually. AWS said the role already exists — which actually meant it was fine.
Then we deleted the broken CloudFormation stack from the CloudFormation console
and tried creating the cluster again. It worked.

**Lesson:** Always check CloudFormation for leftover stacks when ECS cluster
creation fails. Terraform and CDK both use CloudFormation under the hood and
failed runs leave stacks behind.

### Task Definition

The blueprint that tells Fargate exactly how to run your container:

- **CPU: 0.25 vCPU** — the minimum Fargate allows. Equivalent to one quarter
  of a CPU core. Very limited — caused slow Spring Boot startup times of 40-60
  seconds in our testing.
- **Memory: 0.5 GB** — minimum for Spring Boot. Anything less and the JVM
  would run out of memory.
- **Image URI** — full ECR path including account ID, region, repository, and tag
- **Port: 8080** — must match `server.port` in `application.properties`
- **Task execution role** — IAM role that allows ECS to pull images from ECR
  and write logs to CloudWatch on behalf of your task

**Logging configuration** — we enabled CloudWatch logging with these settings:
- `awslogs-group: /ecs/springboot-task` — the CloudWatch log group name
- `awslogs-region: ap-south-1` — where to send logs
- `awslogs-stream-prefix: ecs` — prefix for log stream names
- `awslogs-create-group: true` — automatically creates the log group if it
  does not exist

The new AWS console fills these in automatically when you tick Use log collection
and select Amazon CloudWatch. Previously you had to specify each option manually.

### ECS Service

The ECS Service ensures the desired number of tasks are always running. We set
`desiredCount: 1` to keep costs low during testing. In production minimum 2.

**Rolling update deployment strategy:**
- `Min running tasks: 100%` — always keep at least 100% of desired tasks running
- `Max running tasks: 200%` — can temporarily run up to 200% during deployment

During a rolling update ECS starts a new task (now 200% — 2 tasks), waits for
it to pass health checks, then stops the old task (back to 100% — 1 task).
This ensures zero downtime during deployments.

**Deployment circuit breaker** — automatically rolls back if the new deployment
fails health checks repeatedly. Prevents a broken deployment from replacing
a working one. We saw this in action when our first Force new deployment failed
due to slow startup time — the circuit breaker stopped the bad deployment.

## 6. The Complete Request Flow

```
User opens browser → http://springboot-alb-50032053.ap-south-1.elb.amazonaws.com
         ↓
DNS resolves ALB hostname to ALB's public IP addresses
         ↓
Request reaches ALB on port 80 in public subnet
         ↓
ALB checks Target Group for healthy tasks
         ↓
ALB forwards request to healthy Fargate task on port 8080 in private subnet
         ↓
Spring Boot processes request in HelloController.java
         ↓
Returns JSON response: {"status":"ok","message":"Hello from Spring Boot on AWS Fargate"}
         ↓
Response travels back through ALB to user
         ↓
Container logs written to stdout → ECS captures → CloudWatch Log Group
```

## 7. Deployment Update Flow

When we pushed the v2 change:

```
git push origin main
         ↓
GitHub Actions: Maven build → tests → Docker build → push to ECR
New image at: 554716176680.dkr.ecr.ap-south-1.amazonaws.com/springboot-app:latest
         ↓
Manual: ECS Console → Update service → Force new deployment
         ↓
ECS starts new Fargate task with latest image
New task pulls updated image from ECR via NAT Gateway
         ↓
New task starts Spring Boot (takes 40-60 seconds on 0.25 vCPU)
         ↓
ALB health check hits /health every 30 seconds
Once healthy, ALB registers new task and deregisters old task
         ↓
v2 message now showing in browser
```

In production this Force new deployment step is automated — the pipeline runs
`aws ecs update-service --force-new-deployment` after pushing the image.

## 8. What Went Wrong and What We Learned

**Problem 1: Health check failing on new deployment**

The first Force new deployment failed. The task started but ALB marked it
unhealthy before the app finished loading. Health check grace period was 0 seconds —
ALB started checking immediately. Spring Boot took 57 seconds to start on 0.25 vCPU.

Fix: Updated the ECS Service health check grace period to 120 seconds. This tells
ALB to wait 120 seconds before starting health checks — giving the app enough time
to fully initialize.

Lesson: Always set health check grace period higher than your app startup time.
For Spring Boot on low CPU, 120 seconds minimum. On production CPU (0.5+ vCPU)
startup is under 15 seconds and 30 seconds grace period is enough.

**Problem 2: ECS cluster creation failed**

Failed with service linked role error due to a broken CloudFormation stack left
from a previous Terraform experiment. Fix was to delete the broken stack and
recreate the cluster.

Lesson: AWS keeps track of infrastructure changes through CloudFormation.
Failed Terraform or CDK runs leave stacks behind. Always check CloudFormation
when AWS resource creation fails unexpectedly.

## 9. Teardown Order and Why Order Matters

AWS resources have dependencies — you cannot delete a parent resource while
child resources still exist inside it. Deleting in wrong order causes errors.

```
1. ECS Service          ← must stop tasks before deleting cluster
2. ECS Cluster          ← must be empty of services
3. Task Definition      ← deregister then delete
4. Load Balancer        ← must exist before target group can be deleted
5. Target Group         ← delete after ALB
6. NAT Gateway          ← must be deleted before Elastic IP can be released
7. Elastic IP           ← MUST release separately — not auto-deleted with NAT GW
                           leaving it allocated costs money every month
8. VPC                  ← auto-deletes subnets, route tables, internet gateway
                           but only after NAT Gateway is fully deleted
9. ECR Repository       ← delete images first or use force delete
10. CloudWatch Log Group ← independent, can be deleted any time
```

**The Elastic IP step is the most commonly forgotten.** NAT Gateway deletion
does not release the Elastic IP. AWS charges ~$3.60/month for unattached
Elastic IPs. Always verify with:
```bash
aws ec2 describe-addresses --region ap-south-1
```
If this returns any addresses after teardown, release them immediately.

## 10. Production Considerations

**CPU and memory sizing** — 0.25 vCPU caused 40-60 second Spring Boot startup.
Production minimum is 0.5 vCPU for Spring Boot. Properly sized apps start in
under 15 seconds.

**VPC endpoints** — without VPC endpoints, Fargate tasks pull images from ECR
through NAT Gateway → internet → ECR. This has NAT Gateway data processing costs.
With ECR VPC endpoints, traffic stays inside AWS network — faster and cheaper.
Add `com.amazonaws.ap-south-1.ecr.api` and `com.amazonaws.ap-south-1.ecr.dkr`
VPC endpoints in production.

**Auto deployment** — add this step to the pipeline after image push:
```bash
aws ecs update-service \
  --cluster springboot-cluster \
  --service springboot-service \
  --force-new-deployment \
  --region ap-south-1
```

**Multiple tasks across AZs** — production runs minimum 2 tasks, one per
availability zone. If one AZ has issues, the other continues serving traffic.

**Auto scaling** — ECS Service Auto Scaling adjusts task count based on
CloudWatch metrics. Scale up when CPU exceeds 70%, scale down when it drops.
Handles traffic spikes without manual intervention.

**HTTPS** — production ALB uses port 443 with an SSL certificate from AWS
Certificate Manager (ACM). HTTP on port 80 redirects to HTTPS automatically.
Never serve production traffic over plain HTTP.

**Custom domain** — instead of the ALB DNS name, production uses Route 53 to
map a custom domain like `api.yourcompany.com` to the ALB using an Alias record.
